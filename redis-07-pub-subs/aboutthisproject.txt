# Redis Pub/Sub Notification System - Project Explanation

## Project Overview

This project demonstrates how Redis Pub/Sub (Publish/Subscribe) works using:

* Express.js (API Server)
* Redis (Message Broker)
* ioredis (Redis Client)
* Postman (Testing)

The project consists of two main parts:

1. **Publisher** → Sends notifications to Redis.
2. **Subscriber** → Listens for notifications from Redis.

# How Redis Pub/Sub Works

Redis Pub/Sub follows the Publisher-Subscriber pattern.

```text
Publisher
    |
    |
    v
 Redis Channel
    |
    |
    v
Subscribers
```

Instead of sending messages directly to subscribers, the publisher sends messages to a channel.

Any subscriber listening to that channel receives the message instantly.

---

# Components Used

## 1. Express Server

Located in:

```text
index.js
```

Responsibilities:

* Accept HTTP requests
* Create notification payloads
* Publish messages to Redis

---

## 2. Redis

Acts as a message broker.

Responsibilities:

* Receive published messages
* Deliver messages to all active subscribers

Redis does not store Pub/Sub messages.

If no subscriber is listening, the message is lost.

---

## 3. Subscriber

Located in:

```text
subscriber.js
```

Responsibilities:

* Subscribe to a Redis channel
* Receive messages
* Process or display notifications

# Publisher Flow (index.js)

## Step 1: Client Sends Request

```http
POST /notifications
```

Example:

```json
{
  "title": "Server Down"
}
```

---

## Step 2: Create Payload

```js
const payload = {
    title: req.body.title,
    createdAt: new Date().toISOString()
}
```

Generated Payload:

```json
{
  "title": "Server Down",
  "createdAt": "2026-06-18T12:30:00.000Z"
}
```

---

## Step 3: Publish Message

```js
publisher.publish(
    "notifications",
    JSON.stringify(payload)
);
```

Channel:

```text
notifications
```

Message:

```json
{
  "title": "Server Down",
  "createdAt": "2026-06-18T12:30:00.000Z"
}
```

Redis receives the message and forwards it to all active subscribers.

---

## Step 4: Send Response

```js
res.json({
  message: `Notification sent to ${receivers} subscribers!`
});
```

Example Response:

```json
{
  "message": "Notification sent to 2 subscribers!"
}
```

---

# Subscriber Flow (subscriber.js)

## Step 1: Connect to Redis

```js
const subscriber = new Redis(
  process.env.REDIS_URL
);
```

Connection established:

```text
Subscriber → Redis
```

---

## Step 2: Subscribe to Channel

```js
subscriber.subscribe("notifications");
```

The subscriber tells Redis:

```text
I want all messages from:
notifications
```

---

## Step 3: Listen for Messages

```js
subscriber.on(
  "message",
  (channel, message) => {
      console.log(message);
  }
);
```

Whenever Redis receives a message on:

```text
notifications
```

this event is triggered.

---

## Step 4: Parse Message

Messages arrive as strings.

Example:

```text
"{\"title\":\"Server Down\"}"
```

Convert back into an object:

```js
JSON.parse(message);
```

Result:

```js
{
  title: "Server Down"
}
```

# Key Concepts Learned

## Publisher

A publisher sends messages to a Redis channel.

Example:

```js
publisher.publish(...)
```

---

## Subscriber

A subscriber listens to a Redis channel.

Example:

```js
subscriber.subscribe(...)
```

---

## Channel

A logical communication path between publishers and subscribers.

Example:

```text
notifications
```

---

## Broadcast

Every subscriber listening to the channel receives the message.

```text
Publisher
   |
 Redis
   |
---------
|   |   |
A   B   C
```

All receive the same message.

---

# Limitations of Redis Pub/Sub

## No Persistence

If nobody is subscribed:

```text
Publisher → Redis → Lost
```

Message disappears.

---

## No Retry Mechanism

If a subscriber crashes:

```text
Message Lost
```

Redis does not resend it.

---

## Real-Time Only

Subscribers only receive messages published while they are connected.

Old messages cannot be retrieved later.

---

# When to Use Redis Pub/Sub

Good For:

* Live notifications
* Chat systems
* Real-time updates
* Event broadcasting
* WebSocket message distribution

Not Ideal For:

* Email queues
* Payment processing
* Background jobs
* Reliable task execution

For those use cases, use BullMQ or Redis Streams instead.

---

# Interview Summary

This project demonstrates a basic Redis Pub/Sub architecture where an Express API acts as a publisher and sends notifications to a Redis channel. A subscriber process listens to the channel and receives messages in real time. Redis acts as the message broker, enabling decoupled communication between publishers and subscribers through a broadcast mechanism.

# Key Learning

Redis Pub/Sub is:

* Real-time
* Broadcast-based
* Fire-and-forget
* No message persistence
* Every active subscriber receives the message

If a subscriber is offline when a message is published, it will not receive that message later.
