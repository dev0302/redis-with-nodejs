---

Raw Redis List

You write everything yourself.

text
Producer
    │
    ▼
LPUSH emailQueue
    │
    ▼
Redis List
    │
    ▼
Worker
    │
    ├── What if sendMail() fails?
    ├── Retry?
    ├── How many retries?
    ├── Store failed jobs?
    ├── Handle priorities?
    └── Schedule later?

You have to build those features manually.

---

BullMQ

text
Producer
    │
    ▼
BullMQ Queue
    │
    ▼
Redis
    │
    ▼
BullMQ Worker
    │
    ├── Automatic retries
    ├── Delayed execution
    ├── Priorities
    ├── Failure handling
    ├── Concurrency control
    └── Monitoring

BullMQ uses Redis internally but gives you a much richer API.

---

Example

With raw Redis

js
await redis.rpush(
  "emailQueue",
  JSON.stringify(job)
);

Worker:

js
const [, payload] = await redis.blpop("emailQueue", 0);
const job = JSON.parse(payload);

await sendEmail(job);

If sendEmail() throws an error, you must decide what to do.

---

With BullMQ

Producer:

js
await emailQueue.add("send-email", {
  to: "alice@example.com",
  subject: "Welcome",
});

Worker:

js
new Worker("emailQueue", async (job) => {
  await sendEmail(job.data);
});

Enable retries:

js
await emailQueue.add(
  "send-email",
  data,
  {
    attempts: 3,
    backoff: {
      type: "exponential",
      delay: 5000,
    },
  }
);

BullMQ automatically retries if the job fails.

---

My recommendation

If you're learning Redis:

-   ✅ Build one queue yourself using RPUSH + BLPOP (or LPUSH + BRPOP) to understand producer–consumer fundamentals.

If you're building a real application:

-   ✅ Use BullMQ (or a similar mature job queue library).
-   It saves you from reimplementing retries, priorities, delayed jobs, and robust failure handling.

One small correction

For a FIFO queue, the common patterns are:

text
RPUSH + BLPOP   ✅ FIFO

or

text
LPUSH + BRPOP   ✅ FIFO

Using LPUSH + RPOP is also FIFO, but BLPOP/BRPOP (blocking variants) are preferred for workers because they wait efficiently for new jobs instead of polling continuously.
